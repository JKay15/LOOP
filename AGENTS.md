# LOOP Product Repo Root Rules
## Phase 1: Clarification
- In ordinary conversation, the conversation-facing Codex agent itself is the clarification agent before `kernel mode`, must itself perform the clarification reasoning, and must first judge whether the user's desired final effect is already sufficiently clear.
- For repo-local Phase 1 behavior, read `.agents/skills/loop-endpoint-clarification-host/SKILL.md`, allocate the fresh clarification session root through `scripts/new_clarification_session_root.sh`, and reuse the exact same clarification session root only when that same clarification session is still active and already known.
- If any clarifiable point remains, ask exactly one highest-impact clarification question and persist or update the clarification session through `scripts/persist_clarification_question.sh` as the normal `apply-decision` path; Clarification asks only for endpoint-defining facts and must not bundle secondary execution-resource or placeholder questions.
- By default, do not ask implementation-path questions or later-retrievable details such as exact code, file, or asset locations, tooling, working directories, placeholder-versus-final implementation variants, or local asset discovery; if one upstream answer determines downstream choices, ask only that upstream answer first.
- Do not delegate ordinary Phase 1 clarification to a second nested Codex agent, must not merely imitate clarification behavior in free-form conversation, or jump directly into implementation, debugging, planning, repository edits, or command execution before clarification finishes.
- Once the final effect is sufficiently clear, persist that confirmation through `scripts/persist_clarification_confirmation.sh`; if that same user turn already explicitly authorizes proceeding now, say so explicitly and continue directly, otherwise say so explicitly and ask whether to proceed now.
- Only after the user explicitly agrees to proceed, either in that same clarification-completing user turn or in a later reply, may the conversation-facing Codex agent switch into `kernel mode`; once it does, it must explicitly say that it is now in `kernel mode`.
## Phase 2: Kernel Mode Entry
- After entering `kernel mode`, read `docs/contracts/LOOP_SYSTEM_KERNEL_CONTRACT.md`, `docs/contracts/LOOP_SYSTEM_NODE_CONTRACT.md`, and `.agents/skills/child-dispatch/SKILL.md`; for ordinary `loop_product_repo` first-child bootstrap, do not route through generic `loop-mainline`, LeanAtlas mainline docs, or other extra root-level workflow guides.
- Then call the committed endpoint-driven lifecycle bootstrap surface `bootstrap-first-implementer-from-endpoint` through `scripts/bootstrap_first_implementer_from_endpoint.sh` and the repo's locked `uv` environment instead of probing `--help`, using bare `python -m`, reading request schemas in chat, or hand-assembling workspace roots, frozen handoff files, child prompts, or launch glue.
- Let that bootstrap wrapper choose or create the authoritative project folder under `workspace/`; reuse only when authoritative state already points at that exact workspace folder or node ref. Do not pre-create a guessed project folder before the committed endpoint bootstrap wrapper returns the authoritative `workspace_root`. Do not scan old workspace folders, node reports, or evaluator artifacts to guess whether reuse applies.
- During first-child bootstrap, Do not inspect existing external-target files, local asset files, or preflight external publish-target writability during first-child bootstrap. Do not preflight external publish-target writability during first-child bootstrap. If the task needs an external target, freeze one workspace mirror artifact and remember the root kernel owns the final publication step.
- Publication uses the committed external publication helper `scripts/publish_external_from_result.sh`; Publication may happen only after a terminal evaluator-backed result, must not require evaluator PASS before mechanically publishing the exact workspace mirror artifact, must preserve the evaluator verdict honestly, and must use the committed publication helper instead of ad hoc `cp`, `mv`, or Finder automation.
- Same-run evaluator recovery treats `EvaluatorRunState.json` as the durable ledger with the frozen checker graph; `RECOVERY_REQUIRED` is not yet a terminal evaluator-backed result, and `retryable=true` or `status = RECOVERY_REQUIRED` means recovery continues instead of closeout or publication.
- The first implementer LOOP node must use that project folder as its `workspace_root` and current working directory`, and `workspace/AGENTS.md` plus optional `workspace/AGENTS.override.md` are the instruction surface.
- Before child materialization, the root must not directly implement and must first materialize exactly one LOOP node; after that, launch or relaunch only through the committed child launch helper `scripts/launch_child_from_result.sh`, preserve launch policy exactly, must not translate launch_spec into a fresh ad hoc shell command, and must not hand-launch a bare `codex exec` if that would drop the node's launch policy.
- If the implementer reports that the frozen task should split, root-kernel must treat that as a topology proposal that still requires kernel review; root may materialize additional child nodes only after reviewing and accepting an implementer-authored split request, never by silently fanout-launching extra children on its own.
- Prefer the committed supervision helper `scripts/supervise_child_from_launch.sh` while a child remains active or recovery-eligible; do not end the root conversation, do not stop supervision, and do not treat the task as settled just because there is no new status line worth sending yet.
- The child progress snapshot helper `scripts/child_progress_snapshot.sh` is optional observation only: use the child progress snapshot helper when needed, do not force heartbeat-style progress chatter, and rely on document refs observed from committed launch/evaluator logs when reporting whether the child or evaluator actually read useful documentation.
- Until the implementer returns a terminal evaluator-backed result, root work is liveness, status, or runtime-recovery supervision only; the root must not run its own acceptance, testing, inspection, or evaluator-like checking.
- If the task is still unfinished and a child node or evaluator lane hits a strange interruption, treat that as recovery work rather than final reporting, actively attempt recovery, and stop and report only when the interruption has become an explicit non-recoverable blocker outside the kernel's authority or when the task is already complete.
- Stop and report only when the interruption has become an explicit non-recoverable blocker outside the kernel's authority or when the task is already complete.
- If a child stops after a terminal evaluator `FAIL` but the failure is still ordinary in-scope product work, that retryable terminal evaluator `FAIL` is unfinished repair work, not closeout.
- If authoritative state still says a child node is ACTIVE but the backing runtime or session is gone, classify that as orphaned-active recovery only after `scripts/check_child_runtime_status.sh`; quiet child with a still-live PID is supervision work, not recovery.
- When `scripts/check_child_runtime_status.sh` reports `recovery_eligible=true`, use the committed orphaned-active recovery helper `scripts/recover_orphaned_active.sh` as same-node orphaned-active retry, pass direct structured flags to `scripts/recover_orphaned_active.sh`, avoid hand-writing a recovery request JSON file, and do not stall by rereading old workspace artifacts or recovery docs instead of choosing retry or relaunch from durable evidence.
- After same-node orphaned-active recovery returns a canonical launch spec, relaunch through `scripts/launch_child_from_result.sh`; the root must not rebuild retry envelopes, mutate `CODEX_HOME`, or improvise alternate launch variants in chat.
- Product closeout for an auto-split-capable run must report whether split was proposed, whether kernel accepted it, and whether that split materially changed the proof/product outcome; if no split was proposed, say that explicitly instead of leaving the split path implicit.
