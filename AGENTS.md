# LOOP Product Repo Root Rules
## Phase 1: Clarification
- In ordinary conversation, the conversation-facing Codex agent itself is the clarification agent before `kernel mode`, must itself perform the clarification reasoning, and must first judge whether the user's desired final effect is already sufficiently clear.
- For repo-local Phase 1 behavior, read `.agents/skills/loop-endpoint-clarification-host/SKILL.md`, allocate the fresh clarification session root through `scripts/new_clarification_session_root.sh`, and reuse the exact same clarification session root only when that same clarification session is still active and already known.
- If any clarifiable point remains, ask exactly one highest-impact clarification question and persist or update the clarification session through the committed wrapper `scripts/persist_clarification_question.sh` as the normal `apply-decision` path until the final effect is clear or an explicit bypass is returned.
- Clarification asks only for endpoint-defining facts: final effect, observable success criteria, hard constraints, and non-goals.
- By default, do not ask implementation-path questions or later-retrievable details such as exact code, file, or asset locations, tooling, working directories, placeholder-versus-final implementation variants, or local asset discovery.
- Each clarification turn must contain only one unresolved endpoint question; do not bundle a secondary execution-resource or placeholder/fallback question into the same turn.
- If one upstream endpoint answer will determine downstream dependent choices, ask only for that upstream answer first instead of bundling the dependent questions into the same turn.
- If the endpoint needs some asset class or representative asset from a work/domain but not a user-specified exact asset identity, treat exact asset choice, exact local-file choice, and replacement-slot strategy as later implementation rather than Phase 1 clarification.
- Do not delegate ordinary Phase 1 clarification to a second nested Codex agent, must not merely imitate clarification behavior in free-form conversation, must not imitate clarification persistence in free-form conversation, or jump directly into implementation, debugging, planning, repository edits, or command execution before clarification finishes.
- Once the final effect is sufficiently clear, persist that confirmation turn through `scripts/persist_clarification_confirmation.sh`, then say so explicitly and ask whether to proceed now.
## Phase 2: Kernel Mode Entry
- Only after the user explicitly agrees to proceed may the conversation-facing Codex agent switch into `kernel mode`, and it must explicitly say that it is now in `kernel mode`.
- After entering `kernel mode`, read `docs/contracts/LOOP_SYSTEM_KERNEL_CONTRACT.md`, `docs/contracts/LOOP_SYSTEM_NODE_CONTRACT.md`, and `.agents/skills/child-dispatch/SKILL.md`.
- For ordinary `loop_product_repo` first-child bootstrap, those repo-local reads are sufficient; do not route through generic `loop-mainline`, LeanAtlas mainline docs, or other extra root-level workflow guides before calling the committed bootstrap wrapper.
- After those reads, call the committed endpoint-driven lifecycle bootstrap surface `bootstrap-first-implementer-from-endpoint` through the committed wrapper `scripts/bootstrap_first_implementer_from_endpoint.sh` and the repo's locked `uv` environment instead of probing `--help`, using bare `python -m`, rediscovering helper internals by repository search, reading request schemas in chat, or hand-assembling workspace roots, frozen handoff files, child prompts, or launch glue.
- Let the committed endpoint bootstrap wrapper choose or create the authoritative project folder under `workspace/` for ordinary fresh tasks. Reuse a project folder only when authoritative state already points at that exact workspace folder or node ref; otherwise treat the task as fresh and do not adopt an older workspace or child merely because its name or goal looks similar.
- Do not pre-create a guessed project folder before the committed endpoint bootstrap wrapper returns the authoritative `workspace_root`.
- Do not scan old workspace folders, node reports, or evaluator artifacts to guess whether reuse applies.
- If the frozen final effect requires an external publish target outside the project folder, also freeze one workspace mirror artifact path inside the project folder. The implementer LOOP node owns the workspace mirror artifact; the root kernel owns the final publication step.
- Do not inspect existing external-target files, local asset files, or preflight external publish-target writability during first-child bootstrap; Do not preflight external publish-target writability during first-child bootstrap except through the later root publication step after a terminal evaluator-backed result.
- A retryable evaluator-owned `RECOVERY_REQUIRED` result inside the same evaluator run is not yet a terminal evaluator-backed result for task closeout or publication. Publication may happen only after a terminal evaluator-backed result, must not require evaluator PASS before mechanically publishing the exact workspace mirror artifact, and must preserve the evaluator verdict honestly.
- Use the committed external publication helper `scripts/publish_external_from_result.sh` for that final external publication step, and must use the committed publication helper instead of ad hoc `cp`, `mv`, or Finder automation.
- Same-run evaluator recovery must treat `EvaluatorRunState.json` as the durable recovery ledger, preserve the frozen checker graph, and must not authorize a second checker pass that would silently rewrite the current lane plan.
- The first implementer LOOP node must use that authoritative project folder as its `workspace_root` and current working directory, and `workspace/AGENTS.md` plus optional `workspace/AGENTS.override.md` are the instruction surface for implementer project folders under `workspace/`.
- After reading those documents, the root assistant must not directly implement and must first materialize exactly one LOOP node. Root-kernel work before that node is limited to kernel administration: read rules, call the committed bootstrap wrapper, and launch the child through the committed child launch helper.
- After materialization, child launch or relaunch must preserve the node's launch policy exactly, including `workspace_root`, `cwd` / `-C`, `sandbox_mode`, and bounded reasoning; the root kernel must not hand-launch a bare `codex exec` if that would drop the node's launch policy.
- Use the committed child launch helper `scripts/launch_child_from_result.sh` with the exact bootstrap/recovery result ref; it already owns bounded startup-health observation plus retry for retryable early provider transport/capacity failures, reuses an already-live exact same-node child instead of starting a duplicate one, and the root kernel must not translate launch_spec into a fresh ad hoc shell command in chat.
- The committed bootstrap surface owns frozen handoff/prompt generation, including the task-scoped exact evaluator submission/manual/final-effects surface, the authoritative kernel-visible implementer result sink, and any optional workspace-local mirror sink; the root kernel must not hand-edit child prompts or invent alternate sink layouts in chat.
- Until the implementer returns a terminal evaluator-backed result, the root kernel may do liveness, status, or runtime-recovery supervision only. If the evaluator boundary reports `retryable=true` or `status = RECOVERY_REQUIRED`, continue recovery instead of treating that report as final closeout.
- If the task is still unfinished and a child node or evaluator lane hits a strange interruption, provider/runtime/transport failure, session drop, or other nonterminal stop point, treat that as recovery work rather than final reporting and actively attempt recovery.
- If authoritative state still says a child node is ACTIVE but the backing runtime or session is gone, classify that as orphaned-active recovery only after querying the committed child runtime status helper `scripts/check_child_runtime_status.sh`; a quiet child with a still-live PID is supervision work, not recovery.
- If that helper reports `recovery_eligible=true`, treat it as same-node orphaned-active retry unless durable evidence already requires relaunch, pass direct structured flags to `scripts/recover_orphaned_active.sh`, and use that committed orphaned-active recovery helper instead of hand-writing a recovery request JSON file, probing `--help`, using bare `python -m`, reading request schemas in chat, or doing chat-local helper rediscovery; do not stall by rereading old workspace artifacts or recovery docs instead of choosing retry or relaunch from durable evidence.
- Orphaned-active recovery must not hand-build retry envelopes, ad hoc `CODEX_HOME` mutations, or custom launch variants in chat when the committed helper can regenerate the canonical launch spec from durable node state.
- After same-node orphaned-active recovery accepts retry and returns a canonical launch spec, relaunch through `scripts/launch_child_from_result.sh` instead of reconstructing `codex exec` manually.
- Stop and report only when the interruption has become an explicit non-recoverable blocker outside the kernel's authority or when the task is already complete.
- Before the implementer returns a terminal evaluator-backed result, the root kernel must not run its own acceptance, testing, inspection, or evaluator-like checking. If the implementer returns a terminal blocked result, inspect blocker and recovery evidence only; do not replace the evaluator with a root acceptance pass.
