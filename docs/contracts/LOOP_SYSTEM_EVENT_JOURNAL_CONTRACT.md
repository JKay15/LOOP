# LOOP System Event Journal Contract

This contract freezes the current event-journal foundation plus any explicitly labeled compatibility mirrors that are landed ahead of their final canonical milestone.

## Purpose

The event journal is the first durable control-plane event surface introduced during migration. It is not yet the only authority; it starts as a compatibility mirror written from trusted existing authority paths. Some later-milestone concepts may appear here early as compatibility scaffolding, but that does not mean their owning milestone is complete.

## Required behavior

- the journal store must be created deterministically under the runtime root
- deterministic journal meta must initialize once and remain stable across reopen
- journal open/init must fail closed when stored `runtime_root` or `store_format_version` does not match the requested runtime root and supported store format
- journal open/init must fail closed when stored `event_schema_version` does not match the currently supported event schema version
- journal open/init and compatibility reporting must surface the required producer generation used to fence mixed-version legacy writers
- journal open/init and compatibility reporting must also preserve the raw `journal_meta.required_producer_generation` beside the enforced writer-fence truth, so mixed-version rehearsal can distinguish disabled fencing from meta-vs-fence divergence without raw SQLite inspection
- committed events must be appended atomically
- replay of the same logical event identity must be idempotent
- trusted historical accepted-envelope mirroring must preserve an existing durable `envelope_id` when one is already present instead of recomputing current-generation identities, so preserved real-root audit history can be mirrored into committed truth without leaving the original accepted-only gap behind
- trusted historical accepted-envelope mirroring must deterministically deduplicate repeated accepted audit records by `envelope_id` before committing compatibility mirrors, so preserved roots with duplicated audit files do not conflict on replay
- schema version must be recorded on every committed event
- producer generation must be recorded on every committed event
- trusted writers must reject unsupported committed-event schema versions instead of silently mixing old/new semantics in one journal
- trusted writers must reject unsupported producer generations instead of silently mixing fenced and unfenced writers in one journal
- trusted writers and journal open/init must also fail closed when the enforced writer-fence generation is blank/disabled or when raw `journal_meta.required_producer_generation` diverges from the writer-fence truth
- trusted readers/replayers must fail closed when a committed row advertises an unsupported `schema_version`
- the journal boundary itself must reject inserts whose producer generation does not match the required writer-fence generation, so mixed-version legacy writers cannot keep appending new authority once the fence is enabled
- compatibility reporting must treat both disabled writer fencing and raw-meta-vs-writer-fence divergence as incompatible mixed-version states, exposing structured issues and explicit writer-fencing/alignment status instead of silently normalizing them away
