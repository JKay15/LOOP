# LOOP System State Query Contract

This contract freezes `IF-5` for read-only state visibility.

## Purpose

Read-only state queries must stay simple.

## Runtime anchors

- raw state schema: `docs/schemas/LoopSystemKernelState.schema.json`
- authority view schema: `docs/schemas/LoopSystemAuthorityView.schema.json`
- raw state query implementation: `loop_product/kernel/state.py`
- simplified authority view: `loop_product/kernel/query.py`
- commit-state view: `loop_product/kernel/query.py`
- event-journal status view: `loop_product/kernel/query.py`
- operational hygiene view: `loop_product/kernel/query.py`
- runtime liveness view: `loop_product/kernel/query.py`
- projection consistency view: `loop_product/kernel/query.py`

## Required query surfaces

- current authoritative node graph
- node lifecycle state
- current effective requirement set
- budget / complexity view
- blocked reason
- current delegation map
- active evaluator lane visibility
- active child node visibility
- planned child visibility
- accepted-vs-committed-vs-projected crash-boundary visibility
- event-journal meta / commit progress / projection lag visibility
- repo-local heavy-object authority-gap summary visibility through the generic status/projection debug surfaces
- repo-local heavy-object authority-gap summary visibility through the generic commit-state and operational-hygiene debug surfaces
- repo-local heavy-object pressure / cleanup summary visibility through the generic status, hygiene, commit-state, and projection-explanation surfaces
- latest committed operational hygiene / heavy-object observation visibility
- latest committed runtime liveness visibility
- latest runtime reap requested/deferred/confirmed visibility
- projection consistency visibility for committed-seq vs projection-floor debug
- projection explanation visibility for the currently visible bundle's applied and pending committed facts
- projection replay trace visibility for projection-affecting committed truth vs total journal truth and the latest explicit rebuild result
- mixed-version replay-guard trace visibility for one runtime root's compatibility truth across generic replay, projection, status, and commit-state surfaces
- topology command trace visibility for one accepted topology command's audit envelope, canonical/compatibility events, and visible projected node effects
- node lineage trace visibility for one node's visible ancestor chain plus nested launch/lease runtime lineages
- launch lineage trace visibility for one launch identity's canonical launch/identity/supervision facts and visible source-node liveness state
- lease lineage trace visibility for one node's current lease identity through canonical supervision/heartbeat/closure facts and visible source-node liveness state
- runtime identity trace visibility for one node's current process/lease identity through canonical identity, supervision, heartbeat/closure facts, and visible liveness state
- runtime identity parity debugger visibility for one node comparing the trusted current runtime view with the visible runtime-identity trace
- lineage parity debugger visibility for one node comparing the trusted current node/runtime view with the visible node/launch/lease lineage traces
- command parity debugger visibility for one accepted command/result comparing trusted expected effects with currently visible projected truth
- repo-tree cleanup trace visibility for one accepted repo-tree cleanup command's audit envelope, canonical/compatibility events, and committed cleanup receipt
- heavy-object authority-gap audit trace visibility for one accepted repo-root heavy-object audit command's audit envelope, canonical/compatibility events, deterministic audit receipt, and final authority-gap inventory
- heavy-object registration trace visibility for one accepted heavy-object registration command's audit envelope, canonical/compatibility events, and latest matching heavy-object observation
- heavy-object observation visibility for one runtime/object filter's committed duplicate-heavy-tree observations
- heavy-object retention visibility for one heavy-object identity's current committed registration/pin/supersession/gc/reclamation state
- heavy-object pin-release trace visibility for one accepted heavy-object pin-release command's audit envelope, canonical/compatibility events, matching pin anchor, and matching registration
- heavy-object supersession trace visibility for one accepted heavy-object supersession command's audit envelope, canonical/compatibility events, and latest matching registration facts for both the superseded and replacement objects
- projection provenance visibility for the bundle/journal refs that justify the current visible projection
- projection manifest visibility for complete-bundle vs partial/untracked projection surfaces

## Required behavior

- `query_authority_view(...)` must expose the read-only answers directly rather than requiring callers to reconstruct them from raw accepted envelopes
- the simplified authority view must stay read-only and must not mutate kernel state
- the simplified authority view may omit low-level storage internals such as raw `accepted_envelopes` when a direct higher-level answer already exists
- the simplified authority view must keep active child nodes and planned child nodes separate so deferred split state does not masquerade as already-active work
- once kernel accepts deferred activation, the promoted child must leave `planned_child_nodes` and appear in `active_child_nodes` without callers having to inspect raw envelopes
- once kernel accepts deferred merge, the reactivated source must appear active again without forcing callers to infer runtime reactivation from raw envelopes
- trusted read-only query self-heal must route through the explicit projection rebuild boundary rather than inlining a second projection writer path
- `query_commit_state_view(...)` must explain whether current durable state is audit-accepted only, journal-committed but not yet projected, partially projected, or fully projected
- `query_commit_state_view(...)` must also expose the same compact heavy-object authority-gap summary used by the generic status/projection surfaces, including the effective `authority_runtime_root`, so crash-boundary debug remains honest about repo-anchor heavy-object truth without inventing lifecycle events or a second heavy-object authority model
- `query_commit_state_view(...)` must also expose the same compact heavy-object pressure summary used by the generic status/projection surfaces, including repo-anchor `authority_runtime_root`, duplicate-pressure counts, managed sharing-state counts, and reclaimability/blocking counts, without inventing lifecycle events or a second heavy-object cleanup model
- `query_commit_state_view(...)` must also expose one compact `projection_replay_summary` so crash-boundary callers can see total journal seq, projection-affecting committed seq, pending replay mode, explicit repair scope/state/replayability, node-lag state, and the latest rebuild outcome without manually joining multiple replay/debug queries
- `query_commit_state_view(...)` must also expose one compact `crash_consistency_summary` naming the current crash window cell (`accepted_not_committed`, replayable/blocked committed-not-projected, node-lag, repaired-recently, caught-up, or incompatible) so callers do not have to manually join `phase`, replay summary, and latest rebuild-result truth
- `query_commit_state_view(...)` must also expose the same compact `lease_fencing_summary` used by the authority/liveness/generic status surfaces so crash-boundary callers can see whether lease fencing is clean, engaged, drifting, or blocked by incompatible journal truth without manually joining authority and liveness answers
- `query_event_journal_status_view(...)` must surface journal meta, committed seq/count, and current projection lag/debug state without forcing callers to assemble that picture by hand
- `query_event_journal_status_view(...)` must also surface supported journal versions, supported/required producer generation, writer-fencing status, and structured compatibility issues when damaged journal meta is incompatible, without granting normal trusted journal open semantics
- `query_event_journal_status_view(...)` must also expose a compact heavy-object authority-gap summary derived from the existing authority-gap inventory, including the effective `authority_runtime_root`, current candidate counts, unmanaged-count visibility, strongest duplicate evidence, and a small unmanaged-candidate sample, without inventing lifecycle events or a second heavy-object authority model
- `query_event_journal_status_view(...)` must also expose a compact heavy-object pressure summary derived from existing committed observation, sharing, retention, and GC candidate truth, including the effective `authority_runtime_root`, duplicate-pressure counts, managed sharing-state counts, reference/pin pressure, reclaimability/blocking counts, and small object-ref samples, without inventing lifecycle events or a second heavy-object cleanup model
- `query_event_journal_status_view(...)` must also expose the observed producer generations and any unstamped committed-event count, so mixed-version writer-fence drift is inspectable without raw SQL spelunking
- `query_event_journal_status_view(...)` must also expose one compact `projection_replay_summary` distinguishing total committed journal seq from projection-affecting committed seq, pending replay mode, explicit repair scope/state/replayability, node-lag state, and the latest rebuild outcome
- `query_event_journal_status_view(...)` must also expose one compact `mixed_version_summary` carrying replay-guard state, projection-truth availability, rebuild allowance, supported/required producer generations, and blocking compatibility issue fields/reasons so generic status callers can see when replay/projection truth is intentionally fail-closed under mixed-version incompatibility
- `query_event_journal_status_view(...)` must also expose one compact `crash_consistency_summary` carrying the same crash-window cell and repair outcome truth used by the projection/commit surfaces, so generic status callers can see the current crash boundary without assembling multiple query layers by hand
- `query_event_journal_status_view(...)` must also expose the same compact `lease_fencing_summary` used by the authority/liveness/projection surfaces, including stable counts for authoritative attached truth, expired-attached rejection, stale lower-epoch rejection, fenced-owner exit override, latest non-live dominance, and projection non-live active drift
- that same `mixed_version_summary` and its parent generic surfaces must also preserve `journal_meta_required_producer_generation` plus one writer-fence alignment status, so mixed-version rehearsal can distinguish disabled fencing from raw meta-vs-fence divergence without raw SQLite inspection
- `query_operational_hygiene_view(...)` must expose the latest committed hygiene and heavy-object observations, along with their seq/event identity, without requiring callers to trawl raw journal rows
- `query_operational_hygiene_view(...)` must also expose latest committed runtime reap requested/deferred/confirmed facts when the housekeeping root is used as the authority anchor
- `query_operational_hygiene_view(...)` must also expose the same compact heavy-object authority-gap summary used by the generic status/projection surfaces, including the effective `authority_runtime_root`, so callers can see unmanaged heavy-object candidates without leaving the standard hygiene debug surface
- `query_operational_hygiene_view(...)` must also expose the same compact heavy-object pressure summary used by the generic status/projection surfaces, so callers can see duplicate-heavy-object pressure and lifecycle-backed cleanup pressure without leaving the standard hygiene debug surface
- `query_runtime_liveness_view(...)` must expose both the raw attachment-state observation and the effective freshness-aware attachment state for the latest committed liveness head of each node
- `query_runtime_liveness_view(...)` must not mutate the raw committed observation payload when lease freshness has expired; expiry changes the derived effective state only
- `query_runtime_liveness_view(...)` must make expired attached claims visible as non-live effective state rather than silently preserving eternal live truth
- `query_runtime_liveness_view(...)` must keep stale lower-epoch attached claims visible in raw per-node history while excluding them from effective live truth when a dominant higher attached epoch already exists for that node
- `query_runtime_liveness_view(...)` may therefore expose separate raw and effective per-node liveness heads
- `query_runtime_liveness_view(...)` must also expose one compact `lease_fencing_summary` carrying stable lease/fencing matrix cells and counts derived from the current raw/effective per-node liveness heads
- for top-level runtime roots, `query_runtime_liveness_view(...)` may best-effort bootstrap or reuse the repo control-plane service through the existing trusted control-plane helper, but must remain read-only with respect to kernel truth itself
- `query_authority_view(...)` may expose a lease-aware per-node `runtime_projection` that preserves raw runtime-state compatibility while also surfacing the effective freshness-aware runtime truth for that node
- `query_authority_view(...)` may therefore distinguish compatibility `active_child_nodes` from lease-aware `truthful_active_child_nodes` and `nonlive_active_child_nodes`
- `query_authority_view(...)` must also expose the same compact `lease_fencing_summary` so authority callers can see the current lease/fencing matrix state without reconstructing it from nested runtime projections or raw liveness heads
- for top-level runtime roots, `query_authority_view(...)` may best-effort bootstrap or reuse the repo control-plane service through the existing trusted control-plane helper, but must remain read-only with respect to kernel truth itself
- for top-level runtime roots, `query_operational_hygiene_view(...)` may best-effort bootstrap or reuse the repo control-plane service through the existing trusted control-plane helper, but must remain read-only with respect to kernel truth itself
- `query_projection_consistency_view(...)` must expose committed seq, raw projection floors, stale/caught-up status, and replayability state without forcing self-heal first
- `query_projection_consistency_view(...)` must also expose `ProjectionManifest.json` metadata and a `visibility_state` that distinguishes untracked, partial, and atomically visible projection bundles
- `query_projection_consistency_view(...)` should expose projection provenance alignment for the current bundle so readers can see whether kernel/node projections point at the same journal and manifest
- `query_projection_explanation_view(...)` must expose which projection source currently explains the visible bundle and summarize the nearby applied/pending committed event facts without forcing callers to assemble the journal history by hand
- `query_projection_explanation_view(...)` must also carry the same compact heavy-object authority-gap summary, including the effective `authority_runtime_root`, so the generic projection debug surface remains honest about repo-anchor heavy-object truth even when the committed event tail itself contains no local heavy-object lifecycle facts yet
- `query_projection_explanation_view(...)` must also carry the same compact heavy-object pressure summary, including the effective `authority_runtime_root`, so the generic projection debug surface remains honest about current duplicate-heavy-object pressure and cleanup truth even when callers stay on the standard projection explanation surface
- `query_projection_explanation_view(...)` must also expose the same compact `projection_replay_summary` used by the consistency/status/commit-state surfaces so bundle-local explanation and generic crash-boundary debug stay aligned, including explicit repair scope/state/replayability
- `query_projection_explanation_view(...)` and `query_projection_replay_trace_view(...)` must also expose the same compact `crash_consistency_summary`, including requested-node-aware repaired-recently truth when a node-scoped rebuild fact is the relevant latest repair evidence
- `query_projection_explanation_view(...)` must also expose the same compact `lease_fencing_summary` used by the authority/liveness/generic status surfaces so projection callers can see whether effective truth is clean, fenced, drifting, or incompatible without leaving the projection explanation surface
- `query_projection_consistency_view(...)` must distinguish projection-affecting committed seq from total journal committed seq so non-projection committed facts such as rebuild-result events do not falsely mark projections stale
- `query_projection_consistency_view(...)` must expose one compact `projection_replay_summary` carrying replay mode, total-vs-projection seq, pending replay counts, lagging-node counts, explicit repair scope/state/replayability, and the latest rebuild outcome
- `query_projection_consistency_view(...)` must also expose one compact `crash_consistency_summary` so compatibility, commit-boundary, replayability, and repair-cleared truth stay aligned on one cell model even before callers inspect richer traces
- `query_projection_consistency_view(...)`, `query_projection_explanation_view(...)`, `query_projection_replay_trace_view(...)`, and `query_commit_state_view(...)` must fail closed under incompatible journal generations or schema/store metadata, exposing `compatibility_status=incompatible`, a shared `mixed_version_summary`, and no synthetic replay/projection truth instead of raising or pretending rebuild/replay remains available
- disabled writer fencing or raw `journal_meta.required_producer_generation` divergence from the enforced writer-fence truth must also count as incompatible journal state for those same generic surfaces
- `query_projection_replay_trace_view(...)` must stay read-only and explain one runtime root's replay/crash state through projection consistency, projection explanation, projection-affecting applied/pending event heads, and the latest canonical `projection_rebuild_performed` fact
- `query_projection_replay_trace_view(...)` must distinguish total committed journal seq from projection-affecting committed seq and must prefer node-scoped rebuild results when a specific node id is requested
- the compact `projection_replay_summary` must distinguish at least: replayable kernel repair, blocked kernel repair, node-scoped repair, and no remaining repair required
- `query_projection_crash_consistency_matrix_view(...)` must stay read-only and bundle generic event-journal status, commit-state, projection consistency, projection explanation, and projection replay trace under one top-level `crash_consistency_summary` so operators can inspect the current crash window and its supporting causal surfaces without manual joining
- `query_event_journal_mixed_version_trace_view(...)` must stay read-only and explain one runtime root's mixed-version compatibility truth through the shared `mixed_version_summary`, generic event-journal status, and the replay/projection/commit surfaces, while explicitly withholding those nested replay/projection truths when compatibility is broken
- `query_event_journal_mixed_version_trace_view(...)` must also expose both the enforced writer-fence required generation and the raw journal-meta required generation, plus one alignment status that distinguishes `aligned`, `meta_mismatch`, and `disabled`
- `query_lease_fencing_matrix_view(...)` must stay read-only and bundle runtime-liveness, authority, event-journal status, commit-state, and projection explanation under one top-level `lease_fencing_summary`, plus per-node matrix rows naming authoritative attached truth, expired-attached rejection, stale lower-epoch rejection, fenced-owner exit override, latest non-live dominance, and projection non-live active drift
- `query_real_root_recovery_rehearsal_view(...)` must stay read-only and compare one staged real-root rehearsal tree against the frozen recovery checkpoint, exposing checkpoint intent, node-snapshot alignment, fake-live regressions, path-rebase leakage counts, current compatibility/projection visibility truth, plus a conservative `continuation_summary` / per-node `continuation_candidates` classification built from the existing activation and recovery review rules rather than a second heuristic system
- when `query_real_root_recovery_rehearsal_view(...)` is pointed at the original checkpoint runtime root instead of an isolated staged copy, it must skip the path-rebasing leakage scan and report `rebasing_scan_state=not_applicable_source_root` rather than paying whole-tree scan cost for a staged-root-only concern
- `query_real_root_recovery_continuation_matrix_view(...)` must stay read-only and bundle the `r59` rehearsal continuation truth with generic mixed-version, crash-consistency, and lease-fencing surfaces under one top-level alignment/readiness answer so operators can see which unfinished nodes are direct recovery candidates, which remain dependency-gated, and which generic hardening blockers still prevent truthful continuation
- when `query_real_root_recovery_continuation_matrix_view(...)` is pointed at the original checkpoint runtime root, it may use a slim generic-surface mode that omits heavy-object summary scans while preserving mixed-version, crash-consistency, and lease-fencing truth needed for guarded real-root recovery preflight
- `query_real_root_recovery_action_safety_matrix_view(...)` must stay read-only and layer per-node action-safety truth on top of the staged `r59` continuation matrix, explicitly distinguishing recoverable-in-principle nodes from `safe_to_continue_now`, dependency-blocked, crash-window-blocked, lease-drift-blocked, incompatibility-blocked, and operator-judgment-required nodes without submitting any real recovery command
- `query_real_root_recovery_rehearsal_trace_view(...)` must stay read-only and explain one accepted truthful `r59` rehearsal action by preserving the accepted mutation payload's preflight action-safety truth, reusing the generic topology-command trace for the executed action, and bundling the current staged-root action-safety matrix so operators can compare why the action was allowed before execution with what the generic hardening surfaces say after execution
- `query_real_root_accepted_audit_mirror_repair_trace_view(...)` must stay read-only and explain one trusted historical accepted-audit mirror repair through the latest canonical `accepted_audit_mirror_repaired` fact plus the current commit-state, crash-consistency, continuation, and action-safety truth, so operators can verify that an `accepted_not_committed` real-root blocker was repaired without inspecting raw audit files or mutating the staged root
- `query_real_root_projection_adoption_repair_trace_view(...)` must stay read-only and explain one trusted historical projection-adoption repair through the latest canonical `projection_adoption_repaired` fact plus the current commit-state, crash-consistency, continuation, and action-safety truth, so operators can verify that a `committed_not_projected` real-root blocker was repaired without replaying topology side effects or mutating the staged root from query code
- `query_topology_command_trace_view(...)` must stay read-only and explain one accepted topology command through accepted-audit evidence, committed canonical/compatibility event facts, and the currently visible projected source/target node state
- `query_topology_command_trace_view(...)` must reuse `query_node_lineage_trace_view(...)` for the currently visible source/target nodes rather than forcing callers to issue separate node-lineage lookups after reading the command trace
- for accepted recovery commands such as `resume` and `retry`, `query_topology_command_trace_view(...)` must also expose the visible source-node recovery effect, including cleared blocked state and current projected runtime-state summary/attachment details
- for accepted `relaunch`, `query_topology_command_trace_view(...)` must expose both the superseded source-node projection effect and the currently visible replacement child through the same generic source/target-node trace surface
- `query_topology_command_trace_view(...)` must surface explicit causal gaps when the accepted audit envelope, canonical topology-command event, compatibility mirror, or visible projected node effects are missing
- `query_node_lineage_trace_view(...)` must stay read-only and explain one visible node through its ordered ancestor chain plus nested launch/lease/runtime-identity trace surfaces when runtime lineage exists
- `query_node_lineage_trace_view(...)` must reuse `query_launch_lineage_trace_view(...)`, `query_lease_lineage_trace_view(...)`, and `query_runtime_identity_trace_view(...)` rather than inventing a second node-runtime lineage model
- `query_node_lineage_trace_view(...)` must surface explicit aggregate causal gaps when the visible node projection is missing, the ancestor chain is broken, or nested launch/lease/runtime-identity traces still have causal gaps
- `query_launch_lineage_trace_view(...)` must stay read-only and explain one launch identity through committed canonical `launch_started` or `launch_reused_existing`, `process_identity_confirmed`, and `supervision_attached` facts plus the currently visible source-node projection and raw/effective runtime-liveness heads
- `query_launch_lineage_trace_view(...)` must expose the current visible raw and effective liveness heads for that node rather than forcing callers to stitch launch identity and liveness history together from separate query surfaces
- `query_launch_lineage_trace_view(...)` must surface explicit causal gaps when launch-origin facts, process-identity facts, supervision facts, or visible runtime-liveness heads are missing
- `query_lease_lineage_trace_view(...)` must stay read-only and explain one node's current visible lease lineage through the dominant visible liveness head plus committed canonical supervision attachment, heartbeat, and any matching lease-closure facts
- `query_lease_lineage_trace_view(...)` must expose the current visible lease anchor (`node_id`, `launch_event_id`, `lease_owner_id`, `lease_epoch`) together with the currently visible source-node projection and raw/effective liveness heads
- `query_lease_lineage_trace_view(...)` must surface explicit causal gaps when a currently attached lease lacks canonical heartbeat evidence, or when the current visible lease closure kind lacks its matching canonical closure fact
- `query_runtime_identity_trace_view(...)` must stay read-only and explain one node's current visible runtime identity through canonical `process_identity_confirmed`, `supervision_attached`, heartbeat/closure facts, plus the visible raw/effective runtime-liveness heads
- `query_runtime_identity_trace_view(...)` must reuse the existing launch/lease lineage surfaces rather than inventing a second runtime-identity authority model, and it must expose explicit process-identity, supervision-identity, and currently-visible runtime-identity anchors
- `query_runtime_identity_trace_view(...)` must surface explicit causal gaps when the currently visible runtime identity lacks its matching canonical heartbeat or closure fact
- `query_runtime_identity_parity_debugger_view(...)` must stay read-only and compare the trusted current runtime identity against the visible `query_runtime_identity_trace_view(...)` answer
- `query_runtime_identity_parity_debugger_view(...)` must reuse the runtime-identity trace and existing authority/liveness answers rather than inventing a second runtime-identity authority model, and it must report explicit mismatches or `trace_gaps_present` instead of silently normalizing divergence away
- `query_lineage_parity_debugger_view(...)` must stay read-only and compare the trusted current node/runtime identity against the visible `query_node_lineage_trace_view(...)` answer plus its nested launch/lease/runtime-identity traces
- `query_lineage_parity_debugger_view(...)` must reuse the existing authority/liveness and lineage trace surfaces rather than inventing a second lineage authority model, and it must report explicit mismatches or `trace_gaps_present` instead of silently normalizing divergence away
- `query_command_parity_debugger_view(...)` must stay read-only and compare the old trusted-path expected effect of one accepted topology/evaluator/terminal/repo-tree-cleanup/heavy-object command with the currently visible projected effect reported by the existing trace/query surfaces
- `query_command_parity_debugger_view(...)` must reuse the committed trace surfaces rather than inventing a second authority model, and it must report explicit mismatches instead of silently normalizing divergence away
- for accepted topology-family commands, `query_command_parity_debugger_view(...)` must distinguish deferred split targets from relaunch replacement targets through explicit expected visible target-status parity, rather than treating target-node ids alone as sufficient
- for accepted heavy-object discovery, observation, registration, pin, supersession, gc-eligibility, and reclamation commands, `query_command_parity_debugger_view(...)` must reuse the existing heavy-object trace and retention/discovery/observation view surfaces and compare the command's settled committed effect against the visible read-only answer, rather than inventing a second heavy-object authority model
- for accepted heavy-object supersession, gc-eligibility, and reclamation commands, `query_command_parity_debugger_view(...)` must also compare any carried `trigger_kind` / `trigger_ref` linkage instead of treating lifecycle identity alone as sufficient parity
- for accepted heavy-object reference commands, `query_command_parity_debugger_view(...)` must reuse the existing heavy-object reference trace and inventory surfaces and compare the command's settled committed effect against the visible read-only answer, rather than inventing a second heavy-object authority model
